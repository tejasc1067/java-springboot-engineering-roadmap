# Database migrations with Flyway

Every demo so far used `ddl-auto: create-drop` — Hibernate built the schema from your `@Entity` classes at startup, then dropped it on shutdown. That's perfect for tests. It's a disaster for production. The moment a real user puts a row in your database, you cannot drop the table to add a column. This topic is about the production answer: **version-controlled SQL migrations** with Flyway.

---

## The `ddl-auto` debate

Spring Boot's `spring.jpa.hibernate.ddl-auto` accepts five values:

| Value | What Hibernate does at startup |
|---|---|
| `none` | Nothing. The schema must already exist. |
| `validate` | Read the entities; check that the existing tables/columns match. Fail startup if anything's missing. |
| `update` | Try to bring the schema up to match the entities (add missing columns, etc.). **Never deletes.** |
| `create` | Drop all entity tables, recreate them, leave them after shutdown. |
| `create-drop` | Same as `create` but also drop on shutdown. |

For development against an in-memory database: `create-drop` is great. The schema is rebuilt every JVM start; you can change the entities freely.

For production: **`validate` only.** Here's why:

- `create-drop` deletes all data on shutdown. Production data is the product. Never.
- `create` deletes all data on startup. Same problem.
- `update` looks safe — "it only adds, never removes" — but it has hidden traps. It cannot rename columns (a rename looks like a drop + add). It cannot add a NOT NULL column to a table with existing rows (the default value problem). It cannot alter constraints. And worst, it makes the schema a **side effect of Java code review** — whoever changes the entity controls the production schema, with no DBA review and no rollback path.

In a real codebase, the production schema is owned by **versioned SQL files** committed to the repo. Flyway runs those files, in order, exactly once per database. JPA's job is reduced to "check the schema matches my entity model" — `validate`.

---

## How Flyway works

Add the dependency:

```xml
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>
```

Spring Boot's auto-configuration takes it from there: at startup, before JPA initializes, Flyway scans `classpath:db/migration/`, reads each `V{number}__{description}.sql` file in version order, and applies the ones that aren't already recorded in the `flyway_schema_history` table.

The demo's migrations:

```text
src/main/resources/db/migration/
├── V1__create_books.sql
├── V2__add_isbn_to_books.sql
└── V3__seed_two_books.sql
```

`V1__create_books.sql`:

```sql
CREATE TABLE books (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title           VARCHAR(200) NOT NULL,
    author          VARCHAR(100) NOT NULL,
    published_year  INTEGER
);
```

`V2__add_isbn_to_books.sql`:

```sql
ALTER TABLE books ADD COLUMN isbn VARCHAR(13);
CREATE UNIQUE INDEX ux_books_isbn ON books (isbn);
```

`V3__seed_two_books.sql`:

```sql
INSERT INTO books (title, author, published_year, isbn) VALUES
    ('Effective Java', 'Joshua Bloch', 2017, '9780134685991'),
    ('Clean Code', 'Robert Martin', 2008, '9780132350884');
```

At startup, Flyway logs:

```text
Migrating schema "PUBLIC" to version "1 - create books"
Migrating schema "PUBLIC" to version "2 - add isbn to books"
Migrating schema "PUBLIC" to version "3 - seed two books"
Successfully applied 3 migrations
```

…and Hibernate (in `validate` mode) checks the `Book` entity matches the schema and reports no issues.

The `flyway_schema_history` table tracks state:

```text
+-------------+-------------+-----------+--------------+---------+----------+
| installed_rank | version   | description | type         | success | execution_time |
+-------------+-------------+-----------+--------------+---------+----------+
| 1           | 1           | create books | SQL         | true    | 18ms     |
| 2           | 2           | add isbn ...| SQL         | true    | 12ms     |
| 3           | 3           | seed two...| SQL         | true    |  8ms     |
+-------------+-------------+-----------+--------------+---------+----------+
```

Every migration is run exactly once per database, ever.

## File-name conventions

```text
V{version}__{description}.sql        <- versioned migration (runs once, in version order)
R__{description}.sql                  <- repeatable (re-runs whenever its hash changes; views, functions)
U{version}__{description}.sql         <- undo (paid Flyway only; reverses a versioned migration)
```

A few rules:

- Two underscores between the version and the description. (`V1__name.sql`, not `V1_name.sql`.)
- Version numbers can be integers (`V1`), dotted (`V1.0.0`), or timestamps (`V20260609_create_books`). Pick one style for the project.
- Descriptions use underscores in place of spaces (`create_books`, not `create books`).
- Files in `db/migration` are picked up by default. Override with `spring.flyway.locations` if you need to.

Once a migration has been applied to *any* environment, **never edit it.** Flyway compares the file's checksum against the recorded one and fails startup if they differ. To fix a mistake, write a new `V{n+1}__` migration that undoes or amends.

---

## Repeatable migrations — for things that get re-applied

Views, stored procedures, functions — code that needs to be the *latest* version, not append-only history — go in repeatable migrations:

```text
R__current_book_view.sql
```

Flyway re-runs an `R__` migration whenever its file checksum changes. There is no version number, and they always run **after** all `V` migrations of the current run.

## Integrating with JPA

The demo's `application.yml`:

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: validate
  flyway:
    enabled: true
    locations: classpath:db/migration
```

Two things matter here:

1. **`ddl-auto: validate`.** JPA does not modify the schema. If the entity model and the migrations diverge, Hibernate fails startup with a clear "missing column" or "type mismatch" error — and the fix is to write a new migration.
2. **Flyway runs before JPA's `EntityManagerFactory` is created.** That ordering is what makes `validate` viable: by the time JPA looks, all migrations have applied.

What Hibernate's `validate` checks:

- Tables for every `@Entity` exist.
- Every mapped column exists.
- Column types are compatible (some slack on length/precision).

What it does *not* check:

- Indexes (Hibernate doesn't know what indexes exist).
- Foreign-key constraints (defined in migrations only).
- Triggers, views, anything not tied to an entity field.

That's fine — the migration SQL is the source of truth, and `validate` is a sanity check, not a schema-compare tool.

## Common pitfalls

- **Editing an applied migration.** Flyway recomputes the checksum and refuses to start. Fix: revert the edit and write a new `V{n+1}__` migration with the desired change.
- **Forgetting `ddl-auto: validate`.** If you leave it on `update`, Hibernate may "helpfully" add columns the migrations don't have, and now your environments drift. Always pin it to `validate` (or `none`) once Flyway owns the schema.
- **Renaming a column without thinking.** `ALTER TABLE books RENAME COLUMN xxx TO yyy` is one migration in PostgreSQL — but if any code is mid-deploy, the old code may try to read the old name. Renames need a deploy-coordination strategy: add new column, dual-write, deploy code that reads the new column, drop the old column.
- **Putting business logic in migrations.** Migrations should be data-shape changes (`ALTER TABLE`, `CREATE INDEX`, `UPDATE` for backfills). Application logic stays in Java. A migration that reads a config service or makes an HTTP call is a smell.
- **Mixing DDL and DML in one transaction.** Some databases (Postgres, H2) auto-commit DDL. If you `CREATE TABLE` and then `INSERT` in the same migration file, the rollback semantics differ from pure DDL. Keep DDL-only and DML-only migrations separate when you can.
- **Forgetting to add `flyway-core` for new modules.** Spring Boot won't add it automatically just because you put files in `db/migration`. The dependency is the trigger.

---

## What Flyway does not do

- **No automatic rollback.** The OSS edition has no `migrate down`. Production rollbacks are forward-only: write a new migration that reverses the previous one. Plan for this in your deploy process.
- **No schema diff.** Flyway doesn't compare your live schema to the entity model. Hibernate's `validate` is the closest thing, and it's coarse.
- **No state for "the application is rolling back to an older code version."** If the old code uses an older entity model, `validate` may fail because new columns it doesn't know about now exist. The fix is forward-compatible entity changes (new columns are nullable until both deploy phases are done).
- **No drift detection.** If someone runs `ALTER TABLE` manually in production, Flyway doesn't notice. The cure is process: nobody touches production schemas except via migrations.

## Try this yourself

1. Add a `V4__add_page_count.sql` that adds a `page_count INTEGER` column. Add a `pageCount` field to `Book`. Re-run — `validate` now matches; everything works.
2. Add the `page_count` column to `Book` *without* the migration. Re-run — Hibernate startup fails with a clear "missing column" error. That is `validate` doing its job.
3. Edit `V1__create_books.sql` (change the title column's max length). Re-run — Flyway refuses to start: "checksum mismatch." Revert the change.
4. Add a `V5__bad_sql.sql` with a deliberate syntax error. Re-run — Flyway reports the error and the application fails to start. Fix it; remove the failed entry from `flyway_schema_history` if necessary (or use `flyway:repair`).
