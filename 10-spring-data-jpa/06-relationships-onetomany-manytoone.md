# Relationships — `@OneToMany` and `@ManyToOne`

Most real applications have data that links to other data. A book has an author. An order has line items. A user has posts. JPA models these links with **relationship annotations** that turn into foreign-key columns and JOIN queries. This topic covers the two relationships that show up in 90% of code — one-to-many and its inverse, many-to-one — and the easy mistakes that make them misbehave.

---

## The problem this solves

You have two entities that need to reference each other. Without JPA you'd write a `INTEGER author_id` column on the `books` table, write JOIN queries by hand, write code to convert each row into `Book` + (separate query for) `Author`. That's a lot of glue.

JPA collapses it into two annotations:

```java
@Entity class Book {
    @ManyToOne Author author;       // one author per book
}

@Entity class Author {
    @OneToMany(mappedBy = "author") List<Book> books;   // many books per author
}
```

Hibernate generates the `author_id` foreign key column on `books`, runs the JOIN when you traverse, and gives you both directions of navigation for free.

---

## How it works

The demo entities (`CODE_EXAMPLES/06-relationships-onetomany-manytoone/relationship-demo/`):

```java
@Entity
@Table(name = "books")
public class Book {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 200)
    private String title;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "author_id", nullable = false)
    private Author author;
}

@Entity
@Table(name = "authors")
public class Author {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 100)
    private String name;

    @OneToMany(mappedBy = "author", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Book> books = new ArrayList<>();
}
```

Two tables get generated:

```sql
create table authors (
    id     bigint generated by default as identity,
    name   varchar(100) not null,
    country varchar(50),
    primary key (id)
);

create table books (
    id              bigint generated by default as identity,
    title           varchar(200) not null,
    published_year  integer,
    author_id       bigint not null,
    primary key (id),
    foreign key (author_id) references authors(id)
);
```

Note where the foreign key sits: on `books`, named `author_id`. That's the **owning side**.

---

## Owning side vs inverse side

Every bidirectional relationship has one owning side (the side with the foreign key column) and one inverse side (the side that doesn't).

| Side | Annotation | Has the FK column? | Sets the relationship? |
|---|---|---|---|
| **Owning** (`Book`) | `@ManyToOne` with `@JoinColumn` | yes (`author_id`) | yes — `book.setAuthor(...)` writes to the FK column |
| **Inverse** (`Author`) | `@OneToMany(mappedBy = "author")` | no | no — adding to `author.books` does **not** by itself update the FK |

`mappedBy = "author"` means: "I am the inverse side. Look at the field named `author` on `Book` — that's where the foreign key lives."

**The owning side is the one that determines the database state.** If you do this:

```java
Author bloch = new Author("Joshua Bloch", "USA");
Book ej = new Book("Effective Java", 2017);
bloch.getBooks().add(ej);            // updates the in-memory list on Author
// ej.setAuthor(bloch) is NOT called
authorRepository.save(bloch);        // INSERT into authors, but ... no INSERT into books?
```

…you get unexpected behavior. The owning side (`Book.author`) is null, so even with cascade, Hibernate has nothing to persist on the books side — or it inserts a book with `author_id = null`, which violates the `NOT NULL` constraint, depending on cascade settings.

**Fix: keep both sides in sync.** The bidirectional helper pattern:

```java
public class Author {
    public void addBook(Book book) {
        books.add(book);
        book.setAuthor(this);    // <-- the line that matters
    }
    public void removeBook(Book book) {
        books.remove(book);
        book.setAuthor(null);
    }
}
```

Now every test in the demo calls `author.addBook(book)` and both sides are correct.

## Cascade — making operations flow

By default, JPA operations on one entity do not flow to its associated entities. If you `entityManager.persist(author)` and `author` has unsaved `Book`s in its list, the books are *not* persisted — you'd have to call `entityManager.persist(book)` for each one too.

`CascadeType` changes that:

| CascadeType | What flows |
|---|---|
| `PERSIST` | `persist(author)` also persists each book |
| `MERGE` | `merge(author)` also merges each book |
| `REMOVE` | `delete(author)` also deletes each book |
| `REFRESH` | `refresh(author)` also refreshes each book |
| `DETACH` | `detach(author)` also detaches each book |
| `ALL` | all of the above |

In the demo, `Author.books` is `cascade = CascadeType.ALL`, which is why `authorRepository.save(bloch)` also inserts two books in the same `save` call.

**Be deliberate with `CascadeType.REMOVE` and `CascadeType.ALL`.** They mean "deleting the author deletes all their books." For a books-and-authors model that may be what you want (no orphan books in the DB). For something like users-and-orders, it almost certainly isn't (deleting a user shouldn't silently delete their order history).

---

## `orphanRemoval = true` — the subtle one

`CascadeType.REMOVE` says: "if I delete the author, also delete its books."
`orphanRemoval = true` says: "if a book is removed from this author's list (and not re-attached to another author), delete it."

They sound similar; the trigger is different.

```java
bloch.removeBook(secondBook);     // removes from in-memory list, sets book.author = null
authorRepository.save(bloch);
// With orphanRemoval = true, Hibernate issues DELETE FROM books WHERE id = ?
// Without it, the book sits in the DB with author_id = null (which would also violate the
// NOT NULL constraint on author_id, so the save fails — unless you let book.author be nullable).
```

Turn on `orphanRemoval` when the child entity has no meaning outside its parent. A book without an author isn't a thing in the demo's domain.

## Querying across the relationship

Derived methods can traverse a relationship using underscore notation:

```java
List<Book> findByAuthor_Name(String authorName);
List<Book> findByAuthor_Country(String country);
```

Spring Data parses `Author_Name` as "the `name` field of the `author` association on `Book`." Hibernate generates a JOIN:

```sql
select b1_0.id, b1_0.author_id, b1_0.published_year, b1_0.title
from books b1_0
join authors a1_0 on b1_0.author_id = a1_0.id
where a1_0.name = ?
```

Without the underscore (`findByAuthorName`), Spring Data also resolves it correctly in this case — but if `Book` had an `authorName` field *and* an `author.name` traversal, the underscore-less name would prefer the direct field. Always use the underscore for cross-table traversal.

## `@ManyToMany` — mentioned, not demoed

For completeness: when two entities have a many-to-many relationship (a `Book` has many `Tag`s, a `Tag` is on many `Book`s), JPA models it with `@ManyToMany` and a join table:

```java
@ManyToMany
@JoinTable(name = "book_tags",
    joinColumns = @JoinColumn(name = "book_id"),
    inverseJoinColumns = @JoinColumn(name = "tag_id"))
private Set<Tag> tags;
```

In practice, the moment the join table needs *any* extra column (when the tag was added, who added it), you should model the join itself as an entity and use two `@ManyToOne` relationships — that's the *intermediate entity* pattern. Plain `@ManyToMany` is fine for tagging and similar; the intermediate-entity pattern is the workhorse for everything else.

## Common pitfalls

- **Forgetting `mappedBy`.** Without it, Hibernate thinks both sides own the relationship and creates a *second*, unused join table. Symptom: an extra table appears in the schema you didn't expect. Fix: add `mappedBy = "author"` on the `@OneToMany`.
- **Updating only the inverse side.** `author.getBooks().add(book)` without `book.setAuthor(author)` leaves the owning side `null`, so the FK column never gets set. Symptom: the book either doesn't persist, or persists with `author_id = null`. Fix: use a bidirectional helper (`addBook`/`removeBook`).
- **Loading `author.getBooks()` outside a transaction.** Lazy collections throw `LazyInitializationException` when accessed outside the session that loaded the parent. Topic 07 covers this in depth.
- **Cascade ALL on every relationship.** Cascade decisions need to be intentional. `CascadeType.ALL` on a `User`-to-`Order` relationship would mean "delete the user, delete all their orders." Probably wrong. Default to *no* cascade; add only the ones you want.
- **Using `EAGER` to dodge `LazyInitializationException`.** Tempting, dangerous. `EAGER` means "every time you load a `Book`, also load its `Author` — and any of Author's eager associations transitively." On a list of 1,000 books this can balloon to thousands of queries. Topic 07 covers the fix (`JOIN FETCH` / `@EntityGraph`).
- **Storing `author.setName(null)` and expecting it to delete the author.** Setting a field to null does nothing to row existence. Use `repository.delete(author)`.
- **`List<Book>` vs `Set<Book>`.** `List` allows duplicates and preserves order; `Set` does not. For a `@OneToMany` modeling a true 1:N (a book has one author; an author has N distinct books), `Set` is more semantically accurate. The demo uses `List` for clarity — both work.

## Try this yourself

1. Remove the bidirectional helper. Replace `bloch.addBook(book)` with just `bloch.getBooks().add(book)` (skipping the `book.setAuthor(bloch)`). Run the tests — watch which ones fail and read the error. The owning-side rule becomes very concrete.
2. Change `CascadeType.ALL` to no cascade on `Author.books`. Re-run. The `cascadeFromAuthorPersistsBooks` test should fail because the books aren't persisted automatically.
3. Set `orphanRemoval = false` (or remove it). Re-run the `orphanRemovalDeletesUnreferencedBook` test — it should fail because the orphaned book isn't deleted.
4. Add a `Tag` entity (`id`, `name`) and a `@ManyToMany Set<Tag> tags` field on `Book`. Inspect the generated schema: `Hibernate: create table books_tags (book_id ..., tag_id ...)`.
