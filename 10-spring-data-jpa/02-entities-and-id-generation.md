# Entities and ID generation

An entity is a Java class that JPA knows how to load from, and save to, a database table. You write the class once, annotate it, and Hibernate handles the SQL. This topic is the foundation everything else in module 10 builds on — get the entity shape right and the rest of the module is mostly about querying and tuning.

---

## The problem this solves

Module 04 mapped a JDBC `ResultSet` row to a `Book` object by hand:

```java
Book b = new Book();
b.setId(rs.getLong("id"));
b.setTitle(rs.getString("title"));
b.setAuthor(rs.getString("author"));
b.setPublishedYear(rs.getInt("published_year"));
```

Every column, every type, every field — written out. Add a column to the table and four files need to change. Change a column type and you find out at runtime when `rs.getInt` throws on a string.

JPA replaces all of that with annotations. You declare *what* the mapping is — class to table, field to column — and Hibernate generates the SELECT, the INSERT, the result-to-object copy, and the schema DDL (during development).

---

## How it works

Here is the entity from `CODE_EXAMPLES/02-entities-and-id-generation/entity-demo/Book.java`:

```java
@Entity
@Table(name = "books")
public class Book {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 200)
    private String title;

    @Column(nullable = false, length = 100)
    private String author;

    @Column(name = "published_year")
    private Integer publishedYear;

    protected Book() {
        // JPA requires a no-arg constructor.
    }

    public Book(String title, String author, Integer publishedYear) {
        this.title = title;
        this.author = author;
        this.publishedYear = publishedYear;
    }

    // getters and setters
}
```

That's it. Hibernate reads those annotations at startup and:

1. Generates the DDL (in development, with `ddl-auto: create-drop`):
   ```sql
   create table books (
       published_year integer,
       id bigint generated by default as identity,
       author varchar(100) not null,
       title varchar(200) not null,
       primary key (id)
   )
   ```
2. Generates the INSERT when you `em.persist(book)`:
   ```sql
   insert into books (author, published_year, title, id) values (?, ?, ?, default)
   ```
3. Generates the SELECT when you `em.find(Book.class, 42L)`:
   ```sql
   select b1_0.id, b1_0.author, b1_0.published_year, b1_0.title from books b1_0 where b1_0.id = ?
   ```

Run the demo with `mvn test` and watch the console — `show-sql: true` in `application.yml` makes every statement visible. **Reading the generated SQL is the single most useful habit when learning JPA.** If you only watch your Java code, JPA feels like magic; if you watch the SQL too, it becomes mechanical.

### The annotations, one by one

| Annotation | What it does |
|---|---|
| `@Entity` | Marks the class as managed by JPA. Without this, JPA ignores the class. |
| `@Table(name = "books")` | Overrides the default table name (which would be `book` — JPA defaults to the unqualified class name). Optional; use it when the table name doesn't match the class name. |
| `@Id` | Marks the primary key field. Required. Every entity must have exactly one `@Id` (or one composite `@IdClass` / `@EmbeddedId` — rare). |
| `@GeneratedValue` | Tells JPA the database will generate the id (rather than you setting it). The `strategy` says how. |
| `@Column(name=..., nullable=..., length=..., unique=...)` | Overrides the default column name, nullability, length, or uniqueness. Optional — defaults work if your field names match column names. |

### Why the no-arg constructor

JPA needs to instantiate the class via reflection when loading rows — and reflection-based instantiation calls the no-arg constructor. Without it, you get a `org.hibernate.InstantiationException: No default constructor for entity` at runtime.

Make the no-arg constructor `protected` (or `package-private`), not `public`. That way application code is pushed toward the real constructor (with the meaningful arguments) and you don't accidentally create half-built `Book` instances.

### Why `Long`, not `long`

The id is declared as `Long` (boxed), not `long` (primitive). Two reasons:

1. Before the entity is persisted, the id is unset — `null` is the natural "unset" value. A primitive `long` defaults to `0`, which is ambiguous.
2. Hibernate uses `id == null` as the signal for "this entity is new and needs an INSERT" (vs. "this is a managed entity that needs an UPDATE"). If id is primitive, that check breaks.

Same reasoning for `Integer publishedYear` rather than `int` — `null` means "unknown year," not "year 0."

## ID generation strategies

`@GeneratedValue(strategy = ...)` accepts four standard values. The choice matters because different databases support different generators, and the wrong choice can cripple batch inserts.

### `GenerationType.IDENTITY`

Uses the database's auto-increment column (H2, MySQL, SQL Server, Postgres `SERIAL`/`IDENTITY`). The id is **assigned by the database at INSERT time**, so Hibernate has to execute the INSERT to learn the id.

Pros: simplest, works on every common database, no extra table or sequence to manage.
Cons: defeats JDBC batch inserts in Hibernate (Hibernate has to round-trip per row to get the assigned id back).

This is the demo's choice. For typical CRUD apps, the simplicity wins.

### `GenerationType.SEQUENCE`

Uses a database sequence (`CREATE SEQUENCE`). Hibernate asks the database for the next value *before* INSERT, then includes it in the SQL.

```java
@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "book_seq")
@SequenceGenerator(name = "book_seq", sequenceName = "book_id_seq", allocationSize = 50)
private Long id;
```

Pros: batches well (allocate 50 ids in one round trip, use locally for the next 50 inserts).
Cons: Postgres and Oracle support sequences natively; MySQL does not. If you ever switch databases, this matters.

### `GenerationType.AUTO`

Let the provider pick. For Hibernate 6+ this is `SEQUENCE` if the database supports sequences, `IDENTITY` otherwise. Sounds friendly; in practice it adds magic to your schema (Hibernate may create a hidden `hibernate_sequence` table). **Pick a concrete strategy** so you know what's in your schema.

### `GenerationType.TABLE`

Uses a separate table as a sequence emulator. Slow, almost never the right answer in 2026. Mentioned for completeness.

### What about UUIDs?

Spring Data adds `@GeneratedValue(strategy = GenerationType.UUID)` for string-id entities. Useful when you need globally-unique ids (across services, across shards) or you don't want sequential ids leaking row-count info to clients. The cost: 16-byte primary keys vs. 8-byte longs — both index size and join performance pay. Pick UUIDs when you have a reason, not by default.

## A small worked example

The demo test (`BookEntityTest.java`) does three things:

```java
@Test
void persistAssignsAnId() {
    Book book = new Book("Effective Java", "Joshua Bloch", 2017);
    assertThat(book.getId()).isNull();

    em.persist(book);
    em.flush();

    assertThat(book.getId()).isNotNull();
}
```

Before `persist`, the id is `null`. After `persist` + `flush`, Hibernate has executed the INSERT and the database has filled in the id (because `IDENTITY` is "database assigns it"). Hibernate copies that id back onto the in-memory object.

```java
@Test
void roundTripThroughTheDatabase() {
    Book saved = new Book("Java Concurrency in Practice", "Brian Goetz", 2006);
    em.persist(saved);
    em.flush();
    em.clear();                              // forget everything in the persistence context

    Book loaded = em.find(Book.class, saved.getId());

    assertThat(loaded).isNotSameAs(saved);   // different Java object
    assertThat(loaded.getTitle()).isEqualTo("Java Concurrency in Practice");
}
```

The `em.clear()` matters. Without it, `em.find()` would return the *same* in-memory object that was just persisted (Hibernate's first-level cache), so the test would not actually prove the round-trip through the database. After `clear()`, the persistence context is empty and `find()` issues a real SELECT.

That `select b1_0.id, b1_0.author, b1_0.published_year, b1_0.title from books b1_0 where b1_0.id = ?` you see in the console is the proof.

## Common pitfalls

- **`@Entity` import is `jakarta.persistence.Entity`, not `javax.persistence.Entity`.** Spring Boot 3 moved to Jakarta EE. Old tutorials use `javax`. Your IDE will autocomplete the wrong one if you have both on the classpath; check the import line. Fix: change to `jakarta.persistence.*`.
- **Forgetting the no-arg constructor.** Get `InstantiationException: No default constructor` at runtime, often in a test. Fix: add a `protected Book() {}`.
- **Marking the id field `final`.** Reflection can't set final fields after construction. JPA needs to set the id back onto the object after the INSERT; final breaks that. Fix: drop `final`.
- **Using `int`/`long` instead of `Integer`/`Long` for the id.** Breaks Hibernate's "is this new?" check. Fix: use the boxed types.
- **Forgetting `@Table(name = "books")` when the table name doesn't match.** Hibernate quietly looks for a `book` (singular) table, doesn't find it, complains about a missing table during DDL validation. Fix: add `@Table(name = ...)`.
- **Picking `GenerationType.AUTO` and being surprised.** Hibernate may pick SEQUENCE and create a `hibernate_sequence` table you didn't expect. Fix: pick `IDENTITY` or `SEQUENCE` explicitly.

## Try this yourself

1. Add a `String isbn` field to `Book` with `@Column(unique = true, length = 13)`. Re-run the test. Inspect the new DDL in the console — note the `unique` constraint Hibernate emits.
2. Change `GenerationType.IDENTITY` to `GenerationType.SEQUENCE` and add `@SequenceGenerator(name="book_seq", sequenceName="book_id_seq", allocationSize=50)`. Re-run. Look at the generated SQL — you should now see `next value for book_id_seq` *before* each insert, not `default` inside the insert.
3. Add a second entity, `Author`, with its own `@Id` and a single `String name` field. Make sure both tables get created at startup (`Hibernate: create table authors ...` and `Hibernate: create table books ...`).
4. Remove the `protected Book()` constructor and re-run. Read the exception message carefully — it tells you exactly which class is missing what.
